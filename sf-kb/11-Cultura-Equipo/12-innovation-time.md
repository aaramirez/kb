## Modelos de Innovation Time

| Modelo | Structure | Ejemplo |
|--------|-----------|---------|
| **20% Time** | 1 día/semana para projects personales | Google, Atlassian |
| **Hack Days** | Días dedicados trimestrales | Airbnb, LinkedIn |
| **Hackathons** | Eventos de 24-48h | GitHub, Facebook |
| **Innovation Sprints** | 1-2 weeks dedicados | Spotify, Basecamp |
| **10% Time** | 4h/semana | More realistic than 20% |

### Google's 20% Rule (Historical)

- Cada ingeniero podía dedicar 20% de su tiempo a projects personales
- Resultados: Gmail, Google News, AdSense, Google Transit
- Actualmente menos formalizado pero el principle de innovation time persiste
- Key: no requería approval, solo reporting

## Designing Innovation Time

### Program Structure

| Aspect | Best Practice |
|--------|--------------|
| **Time allocation** | 10-20% of sprint, or 1 dedicated day/month |
| **Selection criteria** | Impact + learning potential + alignment with goals |
| **Team formation** | Voluntary, cross-functional teams of 2-4 |
| **Demo format** | End of period, present to org |
| **Follow-through** | Path to production for promising projects |

### Innovation Time Selection Criteria

| Criterion | Weight | Description |
|-----------|--------|-------------|
| **Learning potential** | 30% | Does it teach the team something new? |
| **Business relevance** | 25% | Could this benefit the product/company? |
| **Technical challenge** | 20% | Is it interesting enough to motivate? |
| **Feasibility** | 15% | Can we build a prototype in the time? |
| **Team interest** | 10% | Is the team excited about it? |

### Application Template

```markdown
# Innovation Time Project Proposal

## Problem Statement
What problem are we solving or exploring?

## Hypothesis
We believe that [solution] will achieve [outcome].

## Approach
How will we prototype/experiment?

## Success Criteria
What does "done" look like for this time box?

## Resources Needed
- People: [who, how many]
- Time: [hours/days]
- Tools: [any special tools needed]
- Data: [what data access]

## Potential Impact
- Technical: [what we learn]
- Business: [potential value]
```

## Hackathon Format

### Pre-Hackathon (1-2 weeks before)

| Activity | Purpose |
|----------|---------|
| Theme selection | Give direction, not constraint |
| Idea submission | Let people pitch projects |
| Team formation | Match skills and interests |
| Logistics prep | Space, food, swag, prizes |

### Hackathon Timeline

| Phase | Duration | Activity |
|-------|----------|----------|
| **Kickoff** | 30 min | Rules, theme, teams announced |
| **Hacking** | 24-48h | Build prototype |
| **Demo prep** | 2h | Prepare 3-min demo |
| **Demos** | 1-2h | Each team presents |
| **Judging** | 30 min | Winners announced |
| **Follow-up** | 1 week | Decide which projects to pursue |

### Hackathon Judging Criteria

| Criterion | Weight | Description |
|-----------|--------|-------------|
| **Innovation** | 30% | Originality and creativity |
| **Impact** | 25% | Potential business or technical value |
| **Execution** | 25% | Working prototype, quality of build |
| **Presentation** | 10% | Clarity of demo and explanation |
| **Learning** | 10% | What the team learned |

## Innovation Labs

| Aspect | Description |
|--------|-------------|
| **Dedicated team** | Small team (3-5) focused on innovation |
| **Budget** | Dedicated funding for experiments |
| **Time horizon** | 3-6 month projects |
| **Output** | Prototypes, research papers, new products |
| **Governance** | Separate from main product roadmap |

### When to Use Innovation Labs

| Situation | Appropriate? |
|-----------|-------------|
| Explore new market opportunities | ✅ |
| Build proof of concept for new tech | ✅ |
| Replace regular product development | ❌ |
| Avoid investing in core product | ❌ |
| Need breakthrough innovation | ✅ (with core product investment) |

## Measuring Innovation Time ROI

| Metric | Target | How to measure |
|--------|--------|---------------|
| **Projects started** | > 4/quarter per team | Project tracking |
| **Projects deployed** | > 1/quarter | Production deployments |
| **Learning artifacts** | > 2/quarter | Tech talks, docs, blog posts |
| **New tools adopted** | > 2/year | Tooling changes |
| **Retention impact** | Positive correlation | HR data |
| **Innovation pipeline** | Growing ideas backlog | Idea management |

## Anti-Patterns

| Anti-Patrón | Consecuencia |
|-------------|-------------|
| "20% time" without real allocation | Feature work always wins |
| No follow-through on ideas | Demoralizing, wasted effort |
| Innovation time for "pet projects" | Inequity, resentment |
| Innovation without business context | Ivory tower, wasted resources |
| Over-structured, too many rules | Kills creativity |
| No demo / no celebration | No incentive to participate |
